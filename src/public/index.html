How this app works

This console follows the same mental model as outbound sequencers: create a campaign (often starts as a draft), connect your workspace (Slack), build the sequence, import leads, then monitor and launch — except here the sends are SMS.

Quick start
Campaign flow (draft → live)
  1. New campaign — creates a draft; rename it on step 3 (Settings). Use Save draft if Slack is not ready yet.
  2. Steps 1–4 in the panel: leads (CSV), sequence (your texts), settings (Slack + timing), test & log.
  3. Master inbox in the sidebar shows recent SMS across all campaigns; click a row to open that campaign.
Server needs SUPABASE_URL + service role, SMS provider keys, SLACK_SIGNING_SECRET for interactive Slack payloads.
Where data lives

Railway Postgres holds campaign workspace settings and job queue metadata.

Supabase (separate project) stores sms_prospect, campaign_event_log, and related logs — apply the SQL in supabase/schema-reference.sql to your Supabase project.

Slack app checklist
  1. Slack app — bot scopes: chat:write, chat:write.public, channels:history, im:write (reinstall after changes).
  2. Interactivity URL: https://<your-domain>/slack/actions
  3. Paste xoxb- token and channel ID into the campaign form.
Hosting
Railway deploy
  1. Connect the repo; set DATABASE_URL, API keys, SLACK_SIGNING_SECRET, Supabase vars, SMS keys.
  2. Apply migrations/*.sql to Railway Postgres as needed.
  3. Ensure the public domain target port matches the app listen port (often 8080).

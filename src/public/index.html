How this app works

This console is for SMS outreach: sequences, lead lists, preview, pacing, and Supabase-backed prospects. Use the tabs inside a campaign so you are not staring at one endless page.

Quick start
Create and send an SMS sequence
  1. New campaign — pick a name and paste Slack bot token + channel (human review lives there).
  2. Open the campaign card → Sequence tab → create or pick an SMS campaign, set send window and gaps (carrier-safe defaults help).
  3. Leads tab → upload CSV or paste phones, preview merged messages, then enroll.
  4. Overview tab → test-send or watch the SMS log; inbound replies get sentiment and STOP auto-DNC.
Server needs SUPABASE_URL + SUPABASE_SERVICE_ROLE_KEY, SMS provider keys (e.g. SMSMobileAPI), and Slack signing secret for buttons.
Where data lives

Railway Postgres holds campaign workspace settings and job queue metadata.

Supabase (separate project) stores sms_prospect, campaign_event_log, and related logs — apply the SQL in supabase/schema-reference.sql to your Supabase project.

Optional: email & LinkedIn replies
SmartLead & HeyReach (Slack drafts)

If this workspace also handles email or LinkedIn webhooks, expand Email & LinkedIn integrations when creating/editing the campaign and paste the API keys. Copy the webhook URLs from the Integrations tab into those tools so replies surface in Slack for approve/edit.

Pure SMS-only workflows can leave those keys blank.
Slack app checklist
  1. Slack app — bot scopes: chat:write, chat:write.public, channels:history, im:write (reinstall after changes).
  2. Interactivity URL: https://<your-domain>/slack/actions
  3. Paste xoxb- token and channel ID into the campaign form.

Hosting
Railway deploy
  1. Connect the repo; set DATABASE_URL, API keys, SLACK_SIGNING_SECRET, Supabase vars, SMS keys.
  2. Apply migrations/*.sql to Railway Postgres as needed.
  3. Ensure the public domain target port matches the app listen port (often 8080).
